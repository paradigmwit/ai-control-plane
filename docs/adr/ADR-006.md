# ADR-006: Tool Registry & Deterministic Tools

## Status
Approved

## Context
The Execution Engine needs a reliable, deterministic interface to execute tasks defined in an AI-generated plan.

Challenges:

- Tools must be validated before execution
- Tools may vary: API calls, database queries, local calculations
- Inputs and outputs must follow schemas for reproducibility
- Unauthorized or unsafe tools must be blocked
- Cost and resource usage must be tracked

Without a centralized tool registry:

- Planner may reference non-existent tools
- Execution may fail at runtime
- Policies cannot be enforced consistently
- Observability becomes inconsistent

---

## Decision
We will implement a **Tool Registry** that centralizes metadata, interfaces, and execution for all deterministic tools.

### Tool Registry Responsibilities
1. **Metadata Management**
    - Tool name
    - Input schema
    - Output schema
    - Permissions / allowed roles
    - Cost estimates (API calls, tokens)
    - Retry/fallback configuration

2. **Tool Validation**
    - Confirm tool existence before execution
    - Ensure inputs conform to schema
    - Block disallowed tools based on policy

3. **Tool Execution Interface**
    - Provides standardized interface for the Execution Engine:
      ```text
      execute(tool_name: str, input: dict) -> output: dict
      ```
    - Ensures deterministic behavior
    - Handles errors, retries, and fallback mechanisms

4. **Integration with Policy Engine**
    - Policy engine queries tool registry for step-level enforcement
    - Registry ensures tools are allowed for the given workflow

5. **Observability**
    - Logs tool input, output, execution time, retries
    - Supports execution replay

---

## Alternatives Considered

1. **Decentralized tool execution**
    - Pros: Flexible, minimal overhead
    - Cons: Hard to validate, inconsistent, non-deterministic

2. **Planner directly calls tools**
    - Pros: Simple, fewer components
    - Cons: Breaks separation of concerns, risky, hard to enforce policies

3. **Static hardcoded tool mapping**
    - Pros: Deterministic
    - Cons: Not extensible, requires code change to add tools

---

## Consequences

- Execution engine can safely invoke deterministic tools
- Supports auditability and reproducibility
- Enables centralized policy enforcement
- Simplifies adding, modifying, or disabling tools
- Cost estimation per step is available
- Observability is consistent and complete
