# ADR-005: Execution Engine – DAG Resolution & Parallelism

## Status
Approved

## Context
The AI planner generates a structured execution plan (DAG) with multiple steps and dependencies.

Key requirements:

- Steps may depend on other steps
- Independent steps should execute in parallel
- Failures should not compromise the entire workflow
- Execution must be deterministic and replayable
- Observability must capture all inputs, outputs, and metadata

Without a proper execution engine:

- Workflow steps may execute out-of-order
- Parallelism opportunities may be lost
- Failures may propagate unpredictably
- Execution traces may be incomplete

---

## Decision
We will implement an **Execution Engine** with the following characteristics:

1. **DAG Resolution**
    - Builds a graph from the validated execution plan
    - Ensures no cycles exist (plan validator guarantees acyclicity)
    - Computes execution order based on dependencies

2. **Parallel Execution**
    - Executes independent steps concurrently using threads, async tasks, or worker pools
    - Dependent steps wait for predecessor completion
    - Supports batching where applicable to optimize throughput

3. **Retry and Fallback Mechanisms**
    - Each step defines a configurable retry count
    - Failed steps may trigger fallback tools or default values
    - Partial results can propagate if safe, logged in Observability Layer

4. **Deterministic Behavior**
    - Input/output logging ensures replayability
    - Execution order deterministic within DAG constraints
    - Randomized elements (if any) are controlled or seeded

5. **Integration with Observability Layer**
    - Step start/end timestamps
    - Tool inputs and outputs
    - Latency metrics
    - Retry and fallback events
    - Step-level status (success, failed, skipped)

6. **Integration with Policy Engine**
    - Each step is checked against policy before execution
    - Execution engine enforces step-level limits, cost ceilings, and permissions

---

## Alternatives Considered

1. **Linear step execution**
    - Pros: Simple, easy to implement
    - Cons: Inefficient for independent steps, reduces throughput

2. **Planner-driven execution**
    - Pros: Planner controls order
    - Cons: Non-deterministic, harder to enforce policy, difficult to handle retries

3. **Full parallel execution ignoring dependencies**
    - Pros: Maximum speed
    - Cons: Unsafe; steps with dependencies may fail, leading to inconsistent results

---

## Consequences

- Multi-step workflows can run efficiently while respecting dependencies
- Independent steps benefit from parallelism
- Failures are handled gracefully without affecting unrelated steps
- Observability is complete for debugging and auditing
- Execution can be deterministically replayed from stored plan and traces

---

## Implementation Notes (Optional)
- Use an **in-memory DAG** representation for plan execution
- Employ a **thread pool** or **async executor** for parallel steps
- Integrate **retry/fallback policies** at the step level
- Log all execution metadata to Observability Layer
- Ensure execution engine APIs accept only validated plans
