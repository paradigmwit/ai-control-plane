# ADR-008: Retry and Fallback Policies

## Status
Approved

## Context
Execution of workflows can fail due to:

- API errors or timeouts
- Network interruptions
- Invalid inputs or partial data
- Temporary unavailability of external systems

Without systematic retry/fallback mechanisms:

- Workflows fail unpredictably
- Partial results may be lost
- Downstream steps may fail unnecessarily

---

## Decision
Introduce **step-level Retry and Fallback Policies**:

1. **Retry**
    - Each execution step defines a max retry count
    - Retries are triggered on transient failures (network, timeout)
    - Backoff strategy can be linear or exponential

2. **Fallback**
    - Optional fallback tools or default values
    - Used if step continues to fail after retries
    - Ensures dependent steps can continue safely

3. **Observability Integration**
    - Logs retries, failures, and fallback usage
    - Enables debugging and auditing

4. **Policy Enforcement**
    - Policy engine can limit max retries per step or workflow
    - Can block risky fallbacks for sensitive steps

---

## Alternatives Considered

1. **No retries**
    - Pros: Simpler
    - Cons: Fragile workflows; failure-prone

2. **Planner-handled fallback**
    - Pros: Keeps planner in control
    - Cons: Non-deterministic, harder to monitor

---

## Consequences

- Workflows are resilient to transient failures
- Execution trace contains full retry/fallback history
- Partial results can be safely propagated
- Supports enterprise SLAs and reliability requirements
